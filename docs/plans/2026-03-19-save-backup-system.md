# Save Backup System

Automatic save file backup during gameplay, with interactive recovery.

## Problem

FogMod (and SpeedFog) performs deep modding of Elden Ring. A crash during a
modded warp can corrupt the save file (`ER0000.sl2`), making it impossible to
continue the run. Since SpeedFog runs are short (~1 hour), losing even a few
minutes of progress is significant. An automatic backup system mitigates this
by keeping rolling copies of the save file.

## Overview

The system has three components:

1. **Backup daemon**: runs in the background while the game is running, copies
   the save file every N minutes, keeps the M most recent backups.
2. **Recovery script**: interactive console menu to restore a backup.
3. **Configuration file**: optional overrides for save path, interval, and
   backup count.

All files are generated by `ConfigGenerator.cs` and packaged by
`PackagingWriter.cs`, following the existing pattern for launcher scripts.

## Output Structure

```
output/
├── launch_speedfog.bat        # Launches daemon + ModEngine (Windows)
├── recovery.bat               # Wrapper that calls recovery.ps1 (Windows)
├── linux/
│   ├── launch_speedfog.sh     # Launches daemon + ModEngine (Linux/Proton)
│   ├── recovery.sh            # Interactive restore (Linux)
│   └── backup_daemon.sh       # Internal, called by launch_speedfog.sh
├── backups/
│   ├── config.ini             # Optional config overrides
│   ├── backup_daemon.ps1      # Internal, called by launch_speedfog.bat
│   └── recovery.ps1           # Internal, called by recovery.bat
└── ...
```

Design principles for the file layout:
- The root contains only files the Windows player interacts with directly.
- Linux files go in `linux/` since Linux players are more technical.
- Internal scripts (`backup_daemon.*`) are placed where the player does not
  need to find them: `backups/` for Windows, `linux/` for Linux.
- `recovery.bat` is a thin wrapper that calls `recovery.ps1` with
  `-ExecutionPolicy Bypass`, so the player can double-click it without
  needing to configure PowerShell execution policies. The actual `.ps1`
  logic lives in `backups/` alongside the daemon.

**Breaking change**: `launch_speedfog.sh` moves from the output root to
`linux/`. This requires updating:
- `PackagingWriter.cs` "To play" console output (line 68)
- `CLAUDE.md` Commands section references
- Any documentation referencing the root-level path

## Configuration: `backups/config.ini`

```ini
# SpeedFog Save Backup Configuration
# Uncomment and edit to override defaults

# Set to false to disable automatic backups entirely
# enabled=true

# Path to save file (auto-detected if not set)
# save_path=C:\Users\...\AppData\Roaming\EldenRing\...\ER0000.sl2

# Backup interval in minutes (default: 1)
# interval=1

# Number of backups to keep (default: 10)
# max_backups=10
```

The file is always generated (with all values commented out). Both the daemon
and recovery script read it. It lives in `backups/` alongside the other
backup-related files, keeping the output root clean.

Parsing: simple line-by-line. Lines starting with `#` or empty are ignored.
Lines matching `key=value` are parsed. Unknown keys are ignored.

## Save File Detection

The save file is located at:
- **Windows**: `%APPDATA%\EldenRing\<steam_id>\ER0000.sl2`
- **Linux/Proton**: `~/.local/share/Steam/steamapps/compatdata/1245620/pfx/drive_c/users/steamuser/AppData/Roaming/EldenRing/<steam_id>/ER0000.sl2`

The `<steam_id>` varies per player. Auto-detection logic:

1. If `save_path` is set in `backups/config.ini`, use it directly.
2. Otherwise, scan the platform-specific `EldenRing` directory for
   subdirectories containing `ER0000.sl2`.
3. If exactly one match: use it automatically.
4. If zero matches: print an error message explaining how to set `save_path`
   in the ini file, then exit.
5. If multiple matches: print the found paths and ask the user to set
   `save_path` in the ini file, then exit.

## Backup Daemon

### Lifecycle

```
launch_speedfog.bat
  1. Read backups/config.ini; if enabled=false, skip step 2
  2. Start backup_daemon.ps1 in a minimized window (start /min)
  3. Launch ModEngine (game starts)
  4. Script ends (daemon continues independently)

backup_daemon.ps1
  1. Read backups/config.ini
  2. Detect save file path (or exit with error)
  3. Wait phase: poll for eldenring.exe every 5 seconds, up to 5 minutes
     - If timeout: print error, exit
  4. If save file exists: create pre-run backup (backups/pre-run_<timestamp>.sl2)
     If save file does not exist: skip (first session, no save yet)
  5. Backup loop:
     a. Sleep $interval minutes
     b. If eldenring.exe is not running: print summary, exit
     c. If save file does not exist: skip this iteration
     d. Copy save to backups/ER0000_<timestamp>.sl2
     e. If copied file is <1 MB: log warning (possible partial copy)
     f. Delete oldest periodic backups until at most $max_backups remain
```

The wait phase (step 3) is necessary because the daemon starts before
ModEngine has launched the game. The 5-minute timeout covers slow game
startup.

### Backup naming

- Pre-run backup: `pre-run_YYYY-MM-DD_HH.mm.ss.sl2`
- Periodic backups: `ER0000_YYYY-MM-DD_HH.mm.ss.sl2`

Timestamp is the backup creation time (not the save file's modification time).

### Purge logic

Only periodic backups (prefix `ER0000_`) count toward `max_backups`. After
creating a new backup, the daemon deletes the oldest periodic backups until at
most `max_backups` remain. Pre-run backups are never purged automatically.
They accumulate across runs, but at ~26 MB each this is acceptable for the
short-lived nature of SpeedFog output folders.

### Scope

Only `ER0000.sl2` is backed up. Elden Ring also maintains `ER0000.sl2.bak`
as its own internal backup, but this is not included. The `.sl2` file alone
is sufficient for recovery.

### Process detection

- **Windows**: `Get-Process eldenring -ErrorAction SilentlyContinue`
- **Linux**: `pgrep -x eldenring.exe` (via Proton/Wine)

The daemon polls every sleep interval. When the game process is no longer
found, the daemon prints a summary and exits.

### Logging

The daemon logs to both console and `backups/backup.log` (append mode).
On Windows it runs in a minimized window; on Linux as a background process.
The log file helps debug issues reported by users.

```
[SpeedFog Backup] Save file: C:\Users\...\ER0000.sl2
[SpeedFog Backup] Waiting for Elden Ring to start...
[SpeedFog Backup] Game detected.
[SpeedFog Backup] Pre-run backup: pre-run_2026-03-19_14.30.00.sl2
[SpeedFog Backup] Backup daemon started (interval: 1 min, keep: 10)
[SpeedFog Backup] Backup: ER0000_2026-03-19_14.31.00.sl2
[SpeedFog Backup] Backup: ER0000_2026-03-19_14.32.00.sl2 (purged: ER0000_2026-03-19_14.22.00.sl2)
[SpeedFog Backup] WARNING: Backup ER0000_2026-03-19_14.33.00.sl2 is only 0.5 MB (expected ~26 MB)
[SpeedFog Backup] Game exited. Daemon stopping. (12 backups created)
```

### Error handling

- If the save file disappears mid-run (e.g. moved), print a warning and
  continue looping (it may reappear).
- If the copy fails (file locked), print a warning and retry next interval.
  The save file is only locked briefly during Elden Ring autosaves.

## Launcher Modifications

### `launch_speedfog.bat` (Windows)

```batch
@echo off
REM SpeedFog Launcher for Elden Ring
REM Auto-generated - do not edit manually

setlocal

REM Get the directory where this script is located
set SCRIPT_DIR=%~dp0

REM Start backup daemon in background (minimized window)
start "SpeedFog Backup" /min powershell -ExecutionPolicy Bypass -NoProfile -File "%SCRIPT_DIR%backups\backup_daemon.ps1"

REM Launch ModEngine with our config
"%SCRIPT_DIR%ModEngine\modengine2_launcher.exe" -t er -c "%SCRIPT_DIR%config_speedfog.toml"

endlocal
```

### `launch_speedfog.sh` (Linux)

```bash
#!/bin/bash
# SpeedFog Launcher for Elden Ring (Linux/Proton)
# Auto-generated - do not edit manually

SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"

# Start backup daemon in background
bash "$SCRIPT_DIR/backup_daemon.sh" &

# Launch ModEngine
wine "$SCRIPT_DIR/../ModEngine/modengine2_launcher.exe" -t er -c "$SCRIPT_DIR/../config_speedfog.toml"
```

Note: the Linux launcher lives in `linux/`, so paths to ModEngine and config
use `../` to reach the output root.

## Recovery Script

### User experience

The player runs `recovery.bat` (Windows) or `linux/recovery.sh` (Linux).
On Windows, `recovery.bat` is a one-line wrapper:

```batch
@powershell -ExecutionPolicy Bypass -NoProfile -File "%~dp0backups\recovery.ps1"
```

```
SpeedFog Save Recovery
======================

Save file: C:\Users\...\AppData\Roaming\EldenRing\76561199058957457\ER0000.sl2

Available backups (newest last):

  [5] pre-run_2026-03-19_14.30.00.sl2  (Pre-run backup)
  [4] ER0000_2026-03-19_14.31.00.sl2
  [3] ER0000_2026-03-19_14.32.00.sl2
  [2] ER0000_2026-03-19_14.33.00.sl2
  [1] ER0000_2026-03-19_14.34.00.sl2
  [0] ER0000_2026-03-19_14.35.00.sl2   (most recent)

Select backup to restore [0]:
Restore ER0000_2026-03-19_14.35.00.sl2? (y/n) [y]:
Restored successfully.
```

- Index 0 = most recent backup. Default selection is 0.
- Default confirmation is y.
- Double-Enter restores the most recent backup.

### Logic

1. Read `backups/config.ini` and detect save path (same logic as daemon).
2. List `backups/*.sl2` sorted oldest-first, assign decremental indices
   (newest = 0).
3. Prompt for index (default 0).
4. Prompt for confirmation (default y).
5. Copy selected backup over the save file.
6. Print success or error message.

### Safety

- The recovery script should warn if `eldenring.exe` is currently running
  ("Warning: Elden Ring appears to be running. Restoring while the game is
  running may not work. Continue? (y/n)").
- No backup of the current save is made before restoring. The current
  (possibly corrupted) save is already covered by the most recent periodic
  backup. Adding another copy would complicate the flow for minimal benefit.

## Code Changes

### `ConfigGenerator.cs`

New methods:
- `WriteBackupDaemonPs1(string outputDir)` - generates `backups/backup_daemon.ps1`
- `WriteBackupDaemonSh(string outputDir)` - generates `linux/backup_daemon.sh`
- `WriteRecoveryPs1(string outputDir)` - generates `backups/recovery.ps1`
- `WriteRecoveryBat(string outputDir)` - generates `recovery.bat` (wrapper)
- `WriteRecoverySh(string outputDir)` - generates `linux/recovery.sh`
- `WriteBackupConfig(string outputDir)` - generates `backups/config.ini`

Modified methods:
- `WriteBatchLauncher(string outputDir)` - add daemon launch before ModEngine,
  read `backups/config.ini` to check `enabled` flag
- `WriteShellLauncher(string outputDir)` - add daemon launch, move to `linux/`
  subfolder, adjust relative paths

### `PackagingWriter.cs`

`WritePackageAsync` calls the new `ConfigGenerator` methods after generating
the existing launcher scripts.

### Directory creation

`PackagingWriter` creates `backups/` and `linux/` directories as part of
packaging, so they exist in the output even before the first run.

Both platforms write backups to `backups/` at the output root. The Linux
daemon resolves this via `$SCRIPT_DIR/../backups/` since it lives in `linux/`.
The recovery scripts use the same path resolution.

## Testing

- Manual testing: generate a run, launch, verify backups appear in `backups/`.
- Manual testing: run `recovery.ps1`, verify restore works.
- The scripts are generated as string literals in C#, so unit testing the
  generation itself is not very useful. Focus on integration testing.
- Edge cases to verify manually:
  - No save file found (error message + ini instructions)
  - Multiple Steam IDs (error message + ini instructions)
  - Game not running when daemon starts (should wait and keep checking)
  - Save file locked during copy (should warn and retry next interval)
